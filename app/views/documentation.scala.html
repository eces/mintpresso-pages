OVERVIEW

We're focused on fast development, and simple deployment.

Mintpresso is really nice especially for the team and developer who:

  • wants to log various numbers growing every seconds.
  • wants to save and load data without coding server side script.
  • needs to do cache as soon as possible without any extra configuration.
  • needs a scalable architecture like Amazon cloud.
  • wants to visualize how many users are active or photos are posted right now.
  • plans to do item/user recommendation as soon as possible. We're the best.
  • plans to operate mission critical system like complex event processing or audit monitoring, predictive analysis.

If you're trying to solve a domain specific problem, check these: Loggly for system log, Mixpanel for customer analytics, and New Relic for server monitoring. However, mintpresso is no.1 at item recommendation.

All you need to do is creating new model(user, post, log, or etc) and linking each models as status(like, view, issue, purchase, or etc). Standardizing your data structure brings a huge positive impact. You can browser any data set in web pages and compose any data mining plans without coding explicit logics. Finally, all your data handling process become comfortable, consistent, responsive, and scalable. Really.

Basic API


Everything is JSON and RESTful

All operations can be done with GET POST PUT DELETE.

All requests by POST. and PUT must set Content-Type: application/json.

All responses from GET must have pure JSON.

All responses from DELETE must not have any body.

Add new model

type: String, json: JsObject

Json object must be wrapped by type. $id is optional, as well as empty string will be treated as nothing like null. If $id is duplicated, the operation is cancelled and return 409 Conflict. You can construct any data form in model, but we recommend less than 2 depths. For example, "urls": [ { "host": "..." } ] is still okay and it's all. As the keyword 'some' is reserved, you can't use 'some' as $id.

Status

This is subject + verb + object

Every status has three parts: subject model, verb string and object model. This is opt for store and express relationship between two models as simple as you can. Let's see examples:

  • User reads Post
  • User views Page
  • User support@@mintpresso.com listens Music
  • User support@@mintpresso.com starred Music "Moon River"
  • User listens { count: 53 } Music 1000
  • User follows User
  • Who follows User support@@mintpresso ?
  • User support@@mintpresso follows whom ?

Because status is consist of s+v+o, you can query to find subject candidates by object and verb. For example, if you want to find who follows you, user(?) + "follow" + user($id) can retreive exactly what you want. $id and $no both can be used for this operations.

Kind of confusing? don't worry, take a breath and see below intuitive APIs we prepared. 





GET  /user/james/read/post/20-ways-to-be-successful

GET  /user/james/read/some/post

Order

In Mintpresso, we called whole data mining process as two steps: Order and Pick up. It's a primitive concept of big or small data processing, you'll be love it.

Basic Concept

You can compose any data you sent to mintpresso at anytime. See presets:

  • Counts all models of certain type. (number of users)
  • Counts all models having json value. (number of active users)
  • Counts all status of certain verb. (listen counts)
  • Counts all status having json value. (number of mobile users)
  • Filters all models by condition. (mark trending posts)
  • Suggests items to a user.
  • Recommends a user to a user.

But all of those plans don't have an actual value change to your data. Because that's a job for pick up does instead. We need to split it up as the following reasons: as many as a number of data processing growing, too many url and sub-scheme(key, column name) for saving temporary values float on new problem and increase complexity. In Mintpresso, one data results can serve one or more ways, and also can affect to other data processing like eco-system. You can export the results by publishing static json url, adding json value to model, adding new model, adding new status and etc by clicking 'New Pickup' in Pages. That's why one data result can affect multiple data.

All model and status is working on publish/subscribe model, its message-driven way has benefits on optimization.
